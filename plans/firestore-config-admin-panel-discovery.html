Firestore Config Admin Panel

Discovery Document — FORGE Sprint
Created: 2026-03-21 | Author: Claude (GA) | Owner: JDM

1. Problem Statement

Firestore is a black box for JDM. The Firebase Console is developer-oriented and unusable for a non-technical CEO. A v1 "Firestore Config" tab was built in the Admin Panel providing visibility into 12 system config collections with a health dashboard. However, CRUD capabilities are inconsistent across every level — some field types have add/edit/delete, others are read-only or missing operations entirely. The feature needs to be completed with uniform CRUD at every level and wiring status indicators so JDM knows which collections are actually connected to the platform vs just sitting in the database.

What wiring status means for the user:

  • FULL STACK — Add a new doc → platform picks it up immediately. Edit a value → takes effect on next load.
  • BACKEND ONLY — API reads/writes it, but no portal UI consumes it yet. Changes are stored correctly but invisible to users.
  • PLACEHOLDER / CODE ONLY — Data sits in Firestore. Nothing reads it until code is written. Useful for pre-staging config before features are built.

9. Audit Notes (Post-Discovery)

  • TRK-513 scope clarification: String array inline edit MUST replace the current window.prompt() pattern which is a forbidden API (hookify-enforced). The new inline editor eliminates the need for browser dialogs entirely.
  • TRK-511 + TRK-512 API detail: Field-level add/delete requires API support — either new endpoints (POST/DELETE on /:collection/:docId/field/:fieldName) or handled via the existing PUT endpoint with field-level merge/delete semantics using Firestore FieldValue.delete().
