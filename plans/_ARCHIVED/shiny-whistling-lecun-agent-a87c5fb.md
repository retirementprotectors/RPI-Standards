# API Integration Research: CRM Data Validation & Enrichment

## Executive Summary

RPI needs 5 categories of data validation/enrichment APIs for ~3,500 client records + ongoing new record validation. The research reveals that **WhitePages Pro already covers 4 of the 5 categories** (address, phone type, phone active status, email validation) through its Identity Check API. The only gap is **frontend address autocomplete** (Google Places), which is a UX feature, not a validation API.

**Bottom line recommendation**: WhitePages Pro (which JDM already pays for) + Google Places Autocomplete = covers everything. Layer in one dedicated email validation service (ZeroBounce or NeverBounce) if WhitePages email validation proves insufficient for bulk hygiene.

## 2. Google Places Autocomplete (Frontend Address Widget)

### What It Does

The "type an address and it auto-completes" dropdown widget. This is a **frontend UX feature**, not a backend validation API. It provides real-time address suggestions as the user types, then fills in structured address fields (street, city, state, zip) when they select a suggestion.

### Two Implementation Options

| Option | How It Works | GAS Compatibility |
|--------|--------------|-------------------|
| **Maps JavaScript API (Autocomplete Widget)** | Load Google Maps JS library in HTML, attach to input field | Works in GAS HtmlService -- load the JS via `<script>` tag in HTML template |
| **Places API (New) REST Endpoint** | Server-side HTTP calls for each keystroke | Works via `UrlFetchApp` but would need custom frontend JS to wire up the UX |

**Recommendation**: Use the **Maps JavaScript API Autocomplete Widget**. It's the standard approach and works directly in GAS HtmlService HTML files. You load the Maps JS library with your API key, create a `google.maps.places.Autocomplete` object attached to an input field, and handle the `place_changed` event to extract structured address components.

### Implementation in GAS HtmlService

```html
<!-- In your HTML template (e.g., APPROVAL_UI.html or ProDashX form) -->
<script src="https://maps.googleapis.com/maps/api/js?key=YOUR_KEY&libraries=places"></script>
<script>
  const autocomplete = new google.maps.places.Autocomplete(
    document.getElementById('address-input'),
    { types: ['address'], componentRestrictions: { country: 'us' } }
  );
  autocomplete.addListener('place_changed', function() {
    const place = autocomplete.getPlace();
    // Extract: street_number, route, locality, administrative_area_level_1, postal_code
    // Populate form fields
  });
</script>
```

### Pricing (Post-March 2025)

- **Free tier**: 10,000 free Autocomplete requests per month per API (Essentials category threshold)
- **Per-request (no session)**: $2.83 per 1,000 requests (~$0.003 each). Each keystroke = 1 request.
- **Per-session**: $17.00 per 1,000 sessions (~$0.017 each). Only charged when user selects a prediction.
- **For RPI's volume**: 10,000 free requests/month is likely sufficient. Even with 100 new clients/month, each typing ~10 characters = 1,000 requests. Well within free tier.

### Key Consideration for GAS

Google Places API requires a GCP project with billing enabled and the Places API activated. RPI already has GCP project `90741179392` (`my-project-rpi-mdj-platform`). Just enable the "Places API" and "Maps JavaScript API" in that project. The API key goes in the HTML template's `<script>` tag -- restrict the key to HTTP referrers matching your GAS web app URLs.

**Security note**: The API key is visible in client-side HTML. Use API key restrictions (HTTP referrer restrictions to `*.googleusercontent.com` for GAS web apps, and API restrictions to only Places API + Maps JS API) to prevent misuse.

## 3. Address Verification APIs

### Does JDM Need a Separate Service?

**If WhitePages Pro API access is confirmed**: Probably not for basic validation. WhitePages Pro returns `is_valid`, `is_active` (receiving mail), residential vs commercial classification, and forwarding detection. That covers 80% of address verification needs for an insurance agency.

**If you need USPS CASS-certified standardization**: Then you need a dedicated service. CASS certification matters for mail delivery (especially if RPI sends physical mail to clients). WhitePages Pro is NOT CASS-certified.

### Options Comparison

| Service | CASS? | Cost (3,500 records) | REST API? | Best For |
|---------|-------|---------------------|-----------|----------|
| **Google Address Validation** | YES | ~$60 (3,500 x $0.017) | YES | Already in Google ecosystem |
| **USPS Address API** | YES | FREE | YES (new OAuth-based) | Free, authoritative source |
| **Smarty (SmartyStreets)** | YES | 250 free/mo, then ~$54/mo | YES | Best accuracy reputation |
| **Lob** | YES | Starting ~$260/mo | YES | Also does direct mail |
| **WhitePages Pro** | NO | Already paying | YES | Already subscribed |

### Detailed Breakdown

#### Google Address Validation API
- **Cost**: $17 per 1,000 requests. For 3,500 = ~$60 one-time. Monthly new records (~50-100) = ~$1-2/month.
- **What it returns**: Standardized address, deliverability verdict (DELIVER, SUSPECT, CANNOT_DELIVER), USPS CASS data (carrier route, DPV confirmation, ZIP+4), geocoding.
- **REST endpoint**: `POST https://addressvalidation.googleapis.com/v1:validateAddress`
- **Rate limit**: 6,000 QPM
- **GAS fit**: Excellent. Same GCP project, `UrlFetchApp.fetch()`, API key in Script Properties.

#### USPS Address API (NEW - Replaces Web Tools)
- **Cost**: FREE (must register at developers.usps.com)
- **CRITICAL**: The old USPS Web Tools APIs **shut down January 25, 2026**. The new REST APIs require OAuth 2.0.
- **What it returns**: Standardized address, ZIP+4, carrier route, delivery point barcode, vacancy indicator.
- **GAS fit**: Moderate. OAuth 2.0 client credentials flow is doable in GAS but requires token management.
- **Limitation**: US addresses only (not a problem for RPI).

#### Smarty (SmartyStreets)
- **Cost**: 250 free lookups/month. Paid plans start ~$54/month for 25,000 lookups.
- **What it returns**: Standardized address, deliverability, residential/commercial classification, vacant/active, ZIP+4, county FIPS.
- **REST API**: Clean, simple. Great documentation.
- **GAS fit**: Excellent. Simple API key auth, `UrlFetchApp.fetch()`.
- **Note**: No overage -- API returns 402 if credits exhausted. Need to monitor usage.

### Recommendation

**Google Address Validation API** is the strongest choice for RPI because:
1. Same GCP project already in use
2. CASS-certified
3. Pay-per-use pricing is perfect for ~3,500 records
4. Monthly cost will be under $2 for ongoing validation
5. Returns USPS standardization + deliverability + geocoding in one call

## 4. Phone Type Detection + Active Status

### Does JDM Need a Separate Service?

**If WhitePages Pro API access is confirmed**: No. WhitePages Pro returns:
- `line_type`: Mobile, Landline, Fixed VOIP, Non-fixed VOIP, Premium, Tollfree, etc.
- `carrier`: Carrier name at MVNO level
- `is_prepaid`: True/False
- `is_valid`: Phone validity / carrier assignment

That covers cell vs landline detection and basic active status.

**If you need real-time "is this number currently active on the network"**: WhitePages Pro checks carrier databases but doesn't do live network pings. For that, you'd need Twilio Lookup or Telnyx.

### Options Comparison

| Service | Line Type? | Active Status? | Cost per Lookup | REST API? |
|---------|-----------|---------------|----------------|-----------|
| **WhitePages Pro** | YES (detailed) | Basic (is_valid) | Included in quota | YES |
| **Twilio Lookup** | YES ($0.008) | YES ($0.01-0.03) | $0.008 - $0.04 | YES |
| **Telnyx Number Lookup** | YES ($0.0015) | Basic (carrier check) | $0.0045 - $0.007 | YES |
| **NumVerify** | YES | Basic | Free 100/mo, then $9.99/mo | YES |
| **Abstract API** | YES | Basic | $19/mo for 1,000 | YES |

### Detailed Breakdown

#### Twilio Lookup v2
- **Line Type Intelligence**: $0.008/lookup (~$28 for 3,500 records)
- **Line Status (active check)**: Tiered pricing, ~$0.01-0.03/lookup
- **Caller Name (CNAM)**: $0.01/lookup
- **Reassigned Number detection**: Available
- **SIM Swap detection**: Custom pricing
- **GAS fit**: Excellent. REST API with Basic auth (Account SID + Auth Token). `UrlFetchApp.fetch()` with auth header.
- **Strength**: Most comprehensive phone intelligence. Can detect if number was recently reassigned to a different person.

#### Telnyx Number Lookup
- **Carrier + Line Type**: $0.0015/lookup + $0.0025 for mobile
- **Caller Name (CNAM)**: Additional $0.003
- **Full lookup (Carrier + CNAM)**: ~$0.0045-0.007/lookup (~$16-25 for 3,500)
- **GAS fit**: Good. REST API with Bearer token auth.
- **Strength**: Cheapest per-lookup pricing.

#### NumVerify
- **Free tier**: 100 lookups/month
- **Paid**: $9.99/month for more lookups
- **Returns**: Validation, country, location, carrier, line type
- **GAS fit**: Good. Simple REST API with API key.
- **Weakness**: Less detailed than Twilio for active status.

### Recommendation

**WhitePages Pro first** (already paying). If the phone data proves insufficient (especially for "is this number still active/connected"), add **Twilio Lookup** as a targeted supplement for the ~$28 one-time cost on 3,500 records. Twilio is the gold standard for phone intelligence and the Reassigned Number detection is particularly valuable for an insurance agency where clients may have changed numbers.

## 5. Email Validation

### Does JDM Need a Separate Service?

**WhitePages Pro Identity Check returns**:
- `is_valid`: Email deliverability
- `is_disposable`: Disposable/temporary email detection
- `is_autogenerated`: Auto-generated email detection
- `email_first_seen_date`: How long the email has existed
- `email_domain_creation_date`: Domain age

**That covers the basics.** However, dedicated email validation services are more thorough -- they do actual SMTP handshakes with the recipient's mail server to confirm the mailbox exists, check for catch-all domains, detect role-based addresses (info@, admin@), and have larger databases of disposable email providers.

**For bulk hygiene on 3,500 records**: A dedicated service is worth it. WhitePages Pro's email check is part of its identity verification (cross-referencing email to person), not a purpose-built email hygiene tool.

### Options Comparison

| Service | Cost (3,500) | Bulk? | Disposable Detection? | REST API? | Free Tier |
|---------|-------------|-------|----------------------|-----------|-----------|
| **ZeroBounce** | $20 (2,000 min) x2 = $40 | YES | YES | YES | 100/mo free |
| **NeverBounce** | $28 (3,500 x $0.008) | YES | YES | YES | None (pay-per-use) |
| **Hunter.io** | $49/mo (1,000 verif) | YES (25K batch) | YES | YES | 50 verif/mo free |
| **Mailgun Optimize** | $49/mo (2,500 verif) | YES | YES | YES | 100/mo free |
| **Abstract API** | $19/mo (1,000 calls) | No | YES | YES | Limited free |

### Detailed Breakdown

#### ZeroBounce
- **Pricing**: $0.01/credit. Minimum purchase 2,000 credits = $20. For 3,500 records = $40 (buy 4,000 credits).
- **Subscription option**: 25,000 credits/month = $99/mo ($79/mo annually)
- **Bulk**: Upload file via API, no size limit. Batch API: up to 200 emails per call.
- **Returns**: Valid, Invalid, Catch-All, Unknown, Spamtrap, Abuse, Do Not Mail, disposable detection
- **GAS fit**: Good. REST API with API key.
- **Strength**: Catches spam traps and abuse addresses. Good for CRM hygiene.

#### NeverBounce
- **Pricing**: $0.008/email for up to 10,000. $0.005 for 10K-100K. For 3,500 = ~$28.
- **Bulk**: Upload via API, process thousands at once.
- **Returns**: Valid, Invalid, Disposable, Accept-All, Unknown
- **GAS fit**: Good. REST API with API key.
- **Strength**: Pay-per-use with no subscription required. Perfect for one-time bulk clean + occasional ongoing.
- **Integrations**: REST API, JavaScript widgets, webhooks.

#### Hunter.io
- **Pricing**: $49/mo for 1,000 verifications ($0.049/verif effective). Verification costs 0.5 credits.
- **Rate limit**: 10 req/sec, 300 req/min
- **Returns**: Valid, Invalid, Disposable, Accept-All, Webmail detection
- **GAS fit**: Good. REST API.
- **Weakness**: Most expensive per-verification. Better known for email finding than validation.

#### Mailgun Optimize
- **Pricing**: $49/mo for 2,500 validations. $99/mo for 5,000.
- **Returns**: deliverable, undeliverable, do_not_send, catch_all, unknown
- **GAS fit**: Good. REST API.
- **Note**: Mailgun is primarily an email sending service. Validation is an add-on.

### Recommendation

**NeverBounce** for the initial bulk clean ($28 for 3,500 records, pay-per-use, no subscription). Then use **ZeroBounce** for ongoing validation if volume justifies the $20 minimum purchase, or continue with NeverBounce at $0.008/email for ad-hoc batches.

For RPI's volume (~50-100 new records/month), you'd spend ~$5-10/month on email validation. NeverBounce's pay-per-use model with no monthly subscription is the most cost-effective.

## Architecture Recommendation

### The Right Answer: Option E (Combination)

Different validation needs live at different layers. Here's where each one goes:

```
LAYER 1: Frontend (ProDashX, Sentinel, RIIMO forms)
  └── Google Places Autocomplete (address widget)
      - Lives in HTML templates
      - Pure client-side JS
      - No server-side component needed

LAYER 2: RAPID_CORE (shared library)
  └── Validation utility functions
      - validateAddress_viaAPI(address) → calls Google AV or WP Pro
      - validatePhone_viaAPI(phone) → calls WP Pro
      - validateEmail_viaAPI(email) → calls WP Pro or NeverBounce
      - enrichContact_viaAPI(name, phone, email, address) → calls WP Pro Identity Check
      - All return structured { success, data } responses
      - API keys stored in Script Properties per-project (not in RAPID_CORE)

LAYER 3: RAPID_IMPORT (bulk validation during import)
  └── Batch validation orchestrator
      - Calls RAPID_CORE validation functions in loops
      - Rate limiting / batching logic
      - Stores validation results in MATRIX (validated_at, validation_score, phone_type, etc.)
      - Used during BoB imports, carrier dumps, GHL syncs

LAYER 4: MCP-Hub (Claude Code intelligence)
  └── Contact validation MCP tools (optional, future)
      - validate_contact, enrich_contact, lookup_phone, etc.
      - Useful for Claude-assisted data quality checks
      - Lower priority -- build after Layers 1-3 are working
```

### Why This Layering?

| Layer | Why Here | Why Not Elsewhere |
|-------|----------|-------------------|
| **Places Autocomplete in frontend** | It's a client-side widget -- must be in HTML | Can't run in server-side GAS code |
| **API wrappers in RAPID_CORE** | Every project needs validation | Don't duplicate in each project |
| **Batch orchestration in RAPID_IMPORT** | Import is where bulk data enters | RAPID_CORE shouldn't own batch logic |
| **MCP tools in MCP-Hub** | Nice-to-have for Claude workflows | Not needed for production data flow |

### Why NOT the other pure options?

- **Option A (RAPID_IMPORT only)**: ProDashX and Sentinel need real-time validation too, not just during import. They'd have to duplicate the API wrapper code.
- **Option B (RAPID_CORE only)**: RAPID_CORE is a library -- it can hold utility functions but shouldn't own batch orchestration or frontend widgets.
- **Option C (MCP-Hub only)**: MCPs are for Claude Code intelligence, not production data pipelines. Users filling out forms don't go through MCP.
- **Option D (Frontends only)**: Frontend apps shouldn't make direct API calls to WhitePages Pro (exposes API keys in client-side code).

### Implementation Priority

| Priority | What | Why | Effort |
|----------|------|-----|--------|
| **P1** | Google Places Autocomplete in ProDashX + Sentinel | Immediate UX win, users love it | 2-3 hours |
| **P2** | WhitePages Pro wrapper in RAPID_CORE | Foundation for all validation | 4-6 hours |
| **P3** | Bulk validation in RAPID_IMPORT | Clean existing 3,500 records | 4-6 hours |
| **P4** | NeverBounce email hygiene (one-time bulk clean) | $28, catches what WP Pro misses | 2-3 hours |
| **P5** | Google Address Validation in RAPID_CORE | Only if sending physical mail | 2-3 hours |
| **P6** | MCP tools in MCP-Hub | Claude-assisted data quality | 4-6 hours (future) |
